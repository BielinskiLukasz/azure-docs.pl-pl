# Azure デプロイ モデル

Azure Platform は移行中です。  Azure を初めて使用する場合も、長い間使用してきた場合も、この移行期間中にプラットフォームに対して行われる重要な変更を理解しておくことは重要です。

すべての Azure リソースでは、次のデプロイ モデルの 1 つまたは両方がサポートされています。

- **リソース マネージャー:** これは Azure リソースの最新の配置モデルです。 新しいリソースのほとんどで、既にこのデプロイ モデルがサポートされており、最終的にはすべてのリソースでサポートされる予定です。   
 
- **クラシック:** このモデルが最も多くの既存の Azure リソースで現在サポートされています。 Azure に追加された新しいリソースでは、このモデルはサポートされません。

各 Azure リソースのドキュメントでは、リソースの作成に使用できるサービス モデルについて詳しく説明します。

## このことが重要な理由 

このことが重要な理由は次のとおりです。

- 使用する Azure Platform の機能は、これら 2 つのモデル間で異なります。  たとえば、リソース マネージャーを使用して作成されたリソースの配置モデル (またはリソース マネージャーのみ) を作成できます [Azure リソース マネージャー テンプレート](resource-group-overview.md/#template-deployment), 、クラシック展開モデルで作成されたリソースのことはできません。
- 個々の Azure リソースの機能や動作は、2 つのモデル間で異なる場合もあれば、どちらか 1 つのモデルにのみ存在する場合もあります。  クラシック展開モデルで作成された仮想マシン間でトラフィックを分散するロードは、たとえば、 *暗黙的な* 仮想マシンは、Azure クラウド サービスのメンバーであり、負荷が自動的にクラウド サービス内の仮想マシン間で分散されたためです。 リソース マネージャーを使用して作成された仮想マシンは、クラウド サービスのメンバーではないと、別の Azure ロード バランサーのリソースがある必要があります *明示的に* 複数の仮想マシン間で分散トラフィックの負荷を作成します。  
- Azure リソースを作成、構成、管理する方法は、これら 2 つのモデル間で異なります。
- 一方のデプロイ モデルを使用して作成されたリソースは、もう一方のデプロイ モデルを使用して作成したリソースと同時に使用できるとは限りません。 たとえば、一方のデプロイ モデルを使用して作成された Azure 仮想マシンは、同じデプロイ モデルを使用して作成された Azure Virtual Network のみに接続できます。    

各デプロイ モデルの基になるのは、各リソースのアプリケーション プログラミング インターフェイス (API) です。   [リソース マネージャー API](https://msdn.microsoft.com/library/azure/dn948464.aspx) リソース マネージャーの配置モデルの [サービス管理 API](https://msdn.microsoft.com/library/azure/ee460799.aspx) クラシック展開モデルにします。 これらの Api と対話するコードを記述する *直接*します。  

IT プロフェッショナルただし、通常 Api とのやり取りこれら *間接的に* web ブラウザーでグラフィカルなポータルを使用して、Windows コンピューターで Azure PowerShell コマンドレットを使用して、または、Windows、OS X または Linux コンピューターで、Azure コマンド ライン インターフェイス (CLI) を使用しています。 IT プロフェッショナルが使用するこれら 3 つの間接的な方法はすべて、API と直接対話します。 つまり、Azure Platform やリソースに新機能が導入されると、必ず最初に API から直接使用できるようになり、間接的な方法では、API が使用可能になった後に、新しいリソースや機能のサポートを受けられるようになります。  

以下のセクションでは、3 つの間接的な方法で異なるデプロイ モデルを使用して Azure リソースを構成する方法について説明します。

## ポータル
Azure には、次の 2 つのポータルがあります。

- **[Azure ポータル](https://manage.windowsazure.com):** をしばらく Azure を使用している場合は、このポータルを使用しました。 従来のデプロイ モデルをサポートする古い Azure リソースを作成および構成する際に使用されます。 このポータルを使用して、リソース マネージャーのみをサポートするリソースを作成および構成することはできません。 
- **[Azure プレビュー ポータル](http://azure.microsoft.com/overview/preview-portal/):** 新しい Azure リソースを使用している場合は、このポータルに使った可能性があります。 このポータルは、一部の Azure リソースの作成および構成に使用できます。 最終的には、このポータルを使用してすべての Azure リソースを作成および構成できるようになります。 両方のデプロイ モデルをサポートする一部のリソースの場合、このポータルを使用すると、いずれかのデプロイ モデルを使用してリソースを作成および構成できます。 

リソースや機能によっては、一方のポータルでしか作成および構成できない場合があります。 また、どちらのポータルでも作成および構成できないリソースや機能もあります。このようなリソースや機能の構成には、PowerShell、CLI、またはその両方を使用できます。 各 Azure リソースのドキュメントでは、リソースの作成に使用できる方法について詳しく説明します。 

## PowerShell
 [PowerShell](powershell-install-configure.md) コマンドラインを使用または作成し、Windows コンピューターから Azure のリソースを構成するためのスクリプトを作成することができます。  個々 の Azure リソースがある [リソース マネージャー コマンドレット](https://msdn.microsoft.com/library/azure/mt125356.aspx), 、[サービス管理コマンドレット](https://msdn.microsoft.com/library/azure/dn708504.aspx), 、またはその両方です。  一部のリソースと機能は、作成や構成に PowerShell または CLI のみを使用できます。 リソースによっては、リソース マネージャーの PowerShell コマンドレットを使用するときに、Azure リソースを作成および構成するためのオプションが 2 つあります。

- **PowerShell コマンドレットの場合のみ:** を作成して、リソースごとに個別にコマンドレットを使用して各 Azure リソースを構成することができます。 これはコマンドラインから実行するか、保存とバージョン管理が可能な PowerShell スクリプトに複数のコマンドを含めることによって実行できます。

- **Azure リソース マネージャー テンプレートと PowerShell コマンドレット:** PowerShell を使用して Azure リソース マネージャー テンプレートを使用して Azure のリソースを作成することができます。 これらのテンプレートは、保存とバージョン管理が可能です。 詳細を読み取って、 [Azure リソース マネージャー テンプレートを使用してアプリケーションを配置](resource-group-template-deploy.md) 記事です。 いくつか [Azure クイック スタート テンプレート](http://azure.microsoft.com/documentation/templates/) の一般的なソリューションをダウンロードしても変更できますが存在します。

## CLI
CLI を使用して、Windows、OS X、または Linux コンピューターから Azure リソースを作成および構成できます。  読み取り、 [Azure CLI をインストール](xplat-cli-install.md) CLI をインストールする、任意のオペレーティング システム上の記事です。 PowerShell のようには、コマンドを使用してリソースを作成しているかどうかに応じて使用する必要がありますが、 [リソース マネージャー](xplat-cli-azure-resource-manager.md) または [クラシック (サービス管理)](virtual-machines-command-line-tools.md) の配置モデルです。
