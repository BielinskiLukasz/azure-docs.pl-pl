# 役割ベースのアクセス制御のトラブルシューティング

## はじめに

[ロール ベースのアクセス制御](../role-based-access-control-configure.md) 強力な機能であり、Azure のリソースに対するきめ細かいアクセスを委任することができます。 つまり、まさに必要な権限を特定の人物に確実に付与できます。 しかし、Azure リソースのリソース モデルが複雑で、どのようなアクセス許可を付与すればよいのか正確に把握しにくい場合もあります。

このドキュメントでは、Azure クラシック ポータルでいくつかのロールを使用する場合に想定されることについて説明します。 すべてのリソースの種類を扱う一般的なロールには次の 3 種類があります。
* 所有者
* 共同作成者
* 閲覧者

所有者と共同作成者には管理操作に対するフル アクセス許可があります。両者の違いは、共同作成者が他のユーザーやグループにアクセス権を付与できないことです。 閲覧者のロールはさらに興味深いので、以下で詳しく考慮します。 [この記事を参照してください](../role-based-access-control-configure.md) 正確にアクセスを許可する方法の詳細。

## App Service ワークロード

### 読み取りアクセス権のみ所有

1 つの Web アプリに対する読み取りアクセス権をユーザーに付与する場合、またはそのアクセス権を付与されているのが自分だけという場合、思いがけず一部の機能が無効になっているという可能性があります。 次の管理機能を必要と **書き込み** (作成者または所有者) は、web アプリへのアクセスし、読み取り専用のシナリオで使用できることはできません。

1. コマンド (開始や停止など)
2. 一般的な構成、スケール設定、バックアップ設定、監視設定などの設定の変更。
3. 発行資格情報およびその他の機密情報 (アプリケーション設定や接続文字列など) へのアクセス。
4. ストリーミング ログ
5. 診断ログの構成
6. コンソール (コマンド プロンプト)
7. アクティブな最新のデプロイ (ローカル Git の継続的デプロイの場合)
8. 所要時間の見積もり
9. Web テスト
10. 仮想ネットワーク (書き込みアクセス権を持つユーザーが仮想ネットワークを事前に構成している場合のみ閲覧者が参照できる)。

これらのいずれかのタイルにアクセスできない場合、Web アプリに対する共同作成者アクセス権が必要です。

### 関連リソースの処理

相互作用する数種類のリソースがあると、Web アプリは複雑になります。 複数の Web サイトが対になっている代表的なリソース グループを以下に示します。

![Web アプリ リソース グループ](./media/role-based-access-control-troubleshooting/website-resource-model.png)

結果として、Web サイトのみに対するアクセス権を付与すると、Web サイト ブレード上の多数の機能が完全に使用できなくなります。

1. これらの項目へのアクセスを必要とする、 **App Service プラン** 、web サイトに対応します。  
    * Web アプリの価格レベル (Free や Standard など) の表示。
    * スケールの構成 (つまり、インスタンスの数、仮想マシンのサイズ、自動スケールの設定)。
    * クォータ (ストレージ、帯域幅、CPU など)。
2. これらのアイテムが全体へのアクセスを必要と **リソース グループ** 、web サイトを格納しています。  
    * SSL の証明書とバインディング (SSL 証明書は同じリソース グループや地理的な場所にあるサイト間で共有できるため)。
    * アラート ルール
    * 自動スケールの設定
    * Application Insights コンポーネント
    * Web テスト

## 仮想マシンのワークロード

Web アプリと同様、仮想マシン ブレード上の機能にも、仮想マシンかリソース グループ内の他のリソースに対する書き込みアクセス権が必要なものがあります。

仮想マシンには以下の関連リソースがあります。
* ドメイン名
* 仮想ネットワーク
* ストレージ アカウント
* アラート ルール

1. これらの項目を必要と **書き込み** 仮想マシンにアクセスします。  
    * Endpoints
    * IP アドレス
    * ディスク
    * 拡張機能
2. これら両方の仮想マシンへの書き込みアクセスを必要とし、 **リソース グループ** であること (とドメイン名)。  
    * 可用性セット
    * 負荷分散セット
    * アラート ルール

これらのいずれかのタイルにアクセスできない場合、管理者に問い合わせてリソース グループに対する共同作成者アクセス権を得る必要があります。
