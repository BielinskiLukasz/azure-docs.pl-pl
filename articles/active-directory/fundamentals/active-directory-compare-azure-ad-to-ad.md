# <a name="compare-active-directory-to-azure-active-directory"></a>Porównywanie usługi Active Directory z usługą Azure Active Directory

Usługa Azure Active Directory to kolejna ewolucja rozwiązań do zarządzania tożsamościami i dostępem dla chmury. Firma Microsoft wprowadziła usługi domenowe Active Directory w systemie Windows 2000, aby dać organizacjom możliwość zarządzania wieloma składnikami i systemami infrastruktury lokalnej przy użyciu jednej tożsamości na użytkownika.

Usługa Azure AD przenosi to podejście na wyższy poziom, udostępniając organizacjom rozwiązanie tożsamości jako usługi (IDaaS) dla wszystkich aplikacji w chmurze i lokalnie.

Większość administratorów IT zna pojęcia usług domenowych Active Directory. W poniższej tabeli przedstawiono różnice i podobieństwa między pojęciami usługi Active Directory a usługą Azure Active Directory.

|Pojęcie|Usługa Active Directory (AD)|Usługa Azure Active Directory |
|:-|:-|:-|
|**Użytkownicy**|||
|Inicjowanie obsługi administracyjnej: użytkownicy | Organizacje ręcznie tworzą użytkowników wewnętrznych lub używają wewnętrznego lub zautomatyzowanego systemu inicjowania obsługi administracyjnej, takiego jak Microsoft Identity Manager, w celu integracji z systemem HR.|Istniejące organizacje ad używają [usługi Azure AD Connect](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-sync-whatis) do synchronizowania tożsamości z chmurą.</br> Usługa Azure AD dodaje obsługę automatycznego tworzenia użytkowników z [systemów hr w chmurze.](https://docs.microsoft.com/azure/active-directory/saas-apps/workday-tutorial) </br>Usługa Azure AD może aprowizować tożsamości w aplikacjach SaaS włączonych w [usłudze SCIM,](https://docs.microsoft.com/azure/active-directory/manage-apps/use-scim-to-provision-users-and-groups) aby automatycznie dostarczać aplikacjom niezbędne szczegóły umożliwiające użytkownikom dostęp. |
|Inicjowanie obsługi administracyjnej: tożsamości zewnętrzne| Organizacje ręcznie tworzą użytkowników zewnętrznych jako zwykłych użytkowników w dedykowanym zewnętrznym lesie usługi AD, co powoduje obciążenie administracyjne związane z zarządzaniem cyklem życia tożsamości zewnętrznych (użytkowników-gości)| Usługa Azure AD zapewnia specjalną klasę tożsamości do obsługi tożsamości zewnętrznych. [Usługa Azure AD B2B](https://docs.microsoft.com/azure/active-directory/b2b/) będzie zarządzać łącze do tożsamości użytkownika zewnętrznego, aby upewnić się, że są prawidłowe. |
| Zarządzanie uprawnieniami i grupy| Administratorzy sprawiają, że użytkownicy są członkami grup. Właściciele aplikacji i zasobów dają grupom dostęp do aplikacji lub zasobów.| [Grupy](https://docs.microsoft.com/azure/active-directory/fundamentals/active-directory-groups-create-azure-portal) są również dostępne w usłudze Azure AD, a administratorzy mogą również używać grup do udzielania uprawnień do zasobów. W usłudze Azure AD administratorzy mogą ręcznie przypisywać członkostwo do grup lub używać kwerendy do dynamicznego dołączania użytkowników do grupy. </br> Administratorzy mogą używać [zarządzania uprawnieniami](https://docs.microsoft.com/azure/active-directory/governance/entitlement-management-overview) w usłudze Azure AD, aby dać użytkownikom dostęp do kolekcji aplikacji i zasobów przy użyciu przepływów pracy i, jeśli to konieczne, kryteriów opartych na czasie. |
| Zarządzanie administratorami|Organizacje będą używać kombinacji domen, jednostek organizacyjnych i grup w udziale w udziale w ad, aby delegować prawa administracyjne do zarządzania katalogiem i zasobami, które kontroluje.| Usługa Azure AD udostępnia [wbudowane role](https://docs.microsoft.com/azure/active-directory/fundamentals/active-directory-users-assign-role-azure-portal) z systemem kontroli dostępu opartego na rolach (RBAC), z ograniczoną obsługą [tworzenia ról niestandardowych](https://docs.microsoft.com/azure/active-directory/users-groups-roles/roles-custom-overview) w celu delegowania uprzywilejowanego dostępu do systemu tożsamości, aplikacji i zasobów, które kontroluje.</br>Zarządzanie rolami można zwiększyć za pomocą [uprzywilejowanego zarządzania tożsamościami (PIM),](https://docs.microsoft.com/azure/active-directory/privileged-identity-management/pim-configure) aby zapewnić dostęp do ról uprzywilejowanych opartych na czasie, ograniczonym czasie lub przepływie pracy. |
| Zarządzanie poświadczeniami| Poświadczenia w usłudze Active Directory są oparte na hasłach, uwierzytelnianiu certyfikatów i uwierzytelnianiu kart inteligentnych. Hasła są zarządzane przy użyciu zasad haseł, które są oparte na długości hasła, wygaśnięcia i złożoności.|Usługa Azure AD używa inteligentnej [ochrony hasłem](https://docs.microsoft.com/azure/active-directory/authentication/concept-password-ban-bad) dla chmury i lokalnie. Ochrona obejmuje inteligentną blokadę oraz blokowanie typowych i niestandardowych fraz haseł i podstawień. </br>Usługa Azure AD znacznie zwiększa bezpieczeństwo [dzięki uwierzytelnianiu wieloskładnikowe](https://docs.microsoft.com/azure/active-directory/authentication/concept-mfa-howitworks) i technologii [bez hasła,](https://docs.microsoft.com/azure/active-directory/authentication/concept-authentication-passwordless) takich jak FIDO2. </br>Usługa Azure AD zmniejsza koszty pomocy technicznej, zapewniając użytkownikom [samoobsługowy system resetowania haseł.](https://docs.microsoft.com/azure/active-directory/authentication/concept-sspr-howitworks) |
| **Aplikacje**|||
| Aplikacje infrastrukturalne|Usługa Active Directory stanowi podstawę dla wielu składników lokalnych infrastruktury, na przykład dostępu do sieci DNS, DHCP, IPSec, WiFi, NPS i VPN|W nowym świecie chmury, Azure AD, jest nowa płaszczyzna kontroli dostępu do aplikacji w porównaniu do polegania na formantów sieciowych. Gdy użytkownicy uwierzytelniają[, Dostęp warunkowy (CA)](https://docs.microsoft.com/azure/active-directory/conditional-access/overview), będzie kontrolować, którzy użytkownicy, będą mieli dostęp do których aplikacji w wymaganych warunkach.|
| Tradycyjne i starsze aplikacje| Większość aplikacji lokalnych używa uwierzytelniania LDAP, uwierzytelniania zintegrowanego z systemem Windows (NTLM i Kerberos) lub uwierzytelniania opartego na nagłówku do kontrolowania dostępu do użytkowników.| Usługa Azure AD może zapewnić dostęp do tych typów aplikacji lokalnych przy użyciu agentów [proxy aplikacji usługi Azure AD](https://docs.microsoft.com/azure/active-directory/manage-apps/application-proxy) działających lokalnie. Za pomocą tej metody usługi Azure AD można uwierzytelniać użytkowników usługi Active Directory w środowisku lokalnym przy użyciu protokołu Kerberos podczas migracji lub konieczności współistnienia ze starszymi aplikacjami. |
| Aplikacje SaaS|Usługa Active Directory nie obsługuje aplikacji SaaS natywnie i wymaga systemu federacyjnego, takiego jak ad fs.|Aplikacje SaaS obsługujące\* uwierzytelnianie OAuth2, SAML i WS można zintegrować w celu użycia usługi Azure AD do uwierzytelniania. |
| Aplikacje biznesowe (LOB) z nowoczesnym uwierzytelnianiem|Organizacje mogą używać usług AD FS z usługą Active Directory do obsługi aplikacji LOB wymagających nowoczesnego uwierzytelniania.| Aplikacje LOB wymagające nowoczesnego uwierzytelniania można skonfigurować do używania usługi Azure AD do uwierzytelniania. |
| Usługi klasy średniej/demona|Usługi działające w środowiskach lokalnych zwykle używają kont usługi AD lub grupowych kont zarządzanych usług (gMSA) do uruchamiania. Te aplikacje odziedziczą uprawnienia konta usługi.| Usługa Azure AD udostępnia [tożsamości zarządzane](https://docs.microsoft.com/azure/active-directory/managed-identities-azure-resources/index) do uruchamiania innych obciążeń w chmurze. Cykl życia tych tożsamości jest zarządzany przez usługę Azure AD i jest powiązany z dostawcą zasobów nie można użyć do innych celów, aby uzyskać dostęp backdoor.|
| **Urządzenia**|||
| Urządzenia przenośne|Usługa Active Directory nie obsługuje natywnie urządzeń przenośnych bez rozwiązań innych firm.| Rozwiązanie firmy Microsoft do zarządzania urządzeniami przenośnymi, Microsoft Intune, jest zintegrowane z usługą Azure AD. Usługa Microsoft Intune udostępnia systemowi tożsamości informacje o stanie urządzenia w celu oceny podczas uwierzytelniania. |
| Komputery stacjonarne z systemem Windows|Usługa Active Directory umożliwia dołączanie do domeny urządzeń z systemem Windows w celu zarządzania nimi przy użyciu zasad grupy, programu System Center Configuration Manager lub innych rozwiązań innych firm.|Urządzenia z systemem Windows można [przyłączać do usługi Azure AD](https://docs.microsoft.com/azure/active-directory/devices/). Dostęp warunkowy może sprawdzić, czy urządzenie jest przyłączone do usługi Azure AD w ramach procesu uwierzytelniania. Urządzeniami z systemem Windows można również zarządzać za pomocą [usługi Microsoft Intune](https://docs.microsoft.com/intune/what-is-intune). W takim przypadku dostęp warunkowy rozważy, czy urządzenie jest skargą (na przykład aktualnymi poprawkami zabezpieczeń i sygnaturami wirusów) przed zezwoleniem na dostęp do aplikacji.|
| Serwery z systemem Windows| Usługa Active Directory zapewnia silne możliwości zarządzania lokalnymi serwerami systemu Windows korzystającymi z zasad grupy lub innych rozwiązań do zarządzania.| Serwerami systemu Windows maszyny wirtualne na platformie Azure można zarządzać za pomocą [usług domenowych usługi Azure AD](https://docs.microsoft.com/azure/active-directory-domain-services/). [Tożsamości zarządzane](https://docs.microsoft.com/azure/active-directory/managed-identities-azure-resources/index) mogą być używane, gdy maszyny wirtualne potrzebują dostępu do katalogu systemu tożsamości lub zasobów.|
| Obciążenia systemu Linux/Unix|Usługa Active Directory nie obsługuje natywnie systemów innych niż Windows bez rozwiązań innych firm, chociaż maszyny z systemem Linux można skonfigurować do uwierzytelniania za pomocą usługi Active Directory jako obszaru Protokołu Kerberos.|Maszyny wirtualne systemu Linux/Unix mogą używać [zarządzanych tożsamości](https://docs.microsoft.com/azure/active-directory/managed-identities-azure-resources/index) w celu uzyskania dostępu do systemu tożsamości lub zasobów. Niektóre organizacje migrują te obciążenia do technologii kontenerów w chmurze, które mogą również używać tożsamości zarządzanych.|
